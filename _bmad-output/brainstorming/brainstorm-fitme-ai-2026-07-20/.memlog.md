- (decision by user) Technique batch: Role Playing, Disney Method, Mind Mapping, Backcasting
- (technique) started Role Playing
- (idea) Nimali (user) wants: usability above all
- (idea) Nimali fears: inaccuracy of nutrition/estimates
- (idea) Nimali non-negotiable: app must be precise when taking inputs
- (idea) Nimali resolves tension: AI may ask a FEW clarifying questions using tappable multiple-choice answers (fast + precise)
- (idea) Dr. Perera: app must NEVER give medical advice (e.g. recommending pills/supplements/medication)
- (idea) Dr. Perera safeguard concern: accuracy of numbers so patients don't crash-diet on bad data
- (idea) Unsafe-target stance: soft guardrail - warn + label 'not recommended' but still let user proceed (respect autonomy, no hard block)
- (idea) Two-tier guardrail: normal unsafe target = yellow warn + proceed; extreme/dangerous case (already underweight, sub-~1000kcal) = RED warning requiring explicit user consent/acknowledgement
- (insight) BEATING HEART of product: effortless food-calorie input + precise nutrition calculation. Everything else is secondary.
- (technique) started Disney Method
- (idea) Dreamer: log a meal by snapping a photo OR typing/speaking natural language - it just becomes a full nutrition breakdown
- (idea) Trust-maker: show the SOURCE of each nutrition value (where the data came from) so numbers feel credible, not invented
- (idea) Trust-maker: during processing show loading time + helpful hints/tips (turns wait into reassurance/education)
- (insight) Realist accuracy engine: decompose composite dish into ingredients + amounts/proportions, then compute nutrition bottom-up from an ingredient-level table (recipe-based calculation)
- (idea) Realist day-one data plan: AI generates ingredient decomposition + nutrition values on the fly (no big hand-curated DB required to launch)
- (insight) Named core risk: AI-generated nutrition values may be inaccurate - accuracy is the recurring make-or-break theme across every role and room
- (decision) KEY: Nutrition source of truth = a PROPER curated ingredient/food nutrition database (not AI-invented numbers). AI's role = parse language + decompose dish + match to DB entries + estimate proportions; DB provides the actual nutrition values = the citable source
- (decision) Tiered nutrition model: 1) curated DB match (trusted, sourced) -> 2) AI on-the-fly estimate when not in DB, clearly marked as ESTIMATED with confidence level; user corrections can promote/refine entries
- (idea) Speed fix: cache DB calls + recently-used/saved foods locally so common logs are instant and work on slow/no network
- (idea) Two lanes clarified: AI smart-path is default when connected; instant no-AI/no-wait path is a graceful fallback on weak/no network using cached DB + recent/saved foods
- (technique) started Mind Mapping
- (idea) Second-most-essential MVP branch = Calorie burning (exercise/activity). Core loop = calories IN (food) vs calories OUT (exercise) -> net calories, which makes food data matter
- (idea) Calorie-out design: baseline burn auto-estimated from profile (BMR + activity level, formula shown); specific workouts entered manually on top. Net calories works even with zero exercise logged
- (decision) MVP SCOPE LINE: AI in v1 = the calorie/nutrition calculation engine + display instructions/guidance. The conversational 'AI assistant' (daily reflection, coaching) is DEFERRED to a later phase. MVP core = food logging + calorie burn + net-calorie dashboard, all powered by AI calc
- (technique) started Backcasting
- (idea) Finish-line success vision: low-effort logging via tappable chips, polished/perfect UX, fully responsive across Android tablet + iPhone + laptop browser (PWA)
- (idea) Backcast step -1 (right before magic moment): a working AI parser + an accurate nutrition DB must already exist
- (insight) Deeper foundation is emotional, not just technical: user attraction + the feeling that 'this is working' + being gently challenged toward her goal drives retention. Progress/motivation must be felt from day 1
- (insight) Motivation mechanic: a beautifully designed summary dashboard (calories in vs burnt, clearly shown) works with or without a set goal - it makes the user REFLECT and decide their next move. Nudge, not force ('make him thinking')
- (decision) FIRST MOVE (step zero, week 1): build user profile creation with her details (name/age/sex/height/weight/activity/goal/units/timezone) - this feeds BMR + targets + the whole dashboard. Foundation before food logging
- (decision) NOT in MVP1 (deferred): manual food search/recently-used/saved meals/custom foods; food-entry review & correction before save; safety ladder guardrails; wearables; coach/nutritionist portal; multilingual/Sinhala; gamification
- (decision) CLARIFIED: MVP1 KEEPS a lightweight 'confirm/edit before save' step (essential for accuracy/trust). What is cut = the heavy manual food-search/saved-meals UI, not the review moment
- (decision) REINSTATED into MVP1: full safety ladder (green/yellow/red consent) for unsafe targets + 'no medical advice' guardrails. Safety is a Must-Have, not deferred
- (decision) MoSCoW SHOULD-HAVE (Phase 2/3): body/weight progress + trend charts; AI daily reflection; weekly report; photo food recognition; daily health check-in (sleep/mood/water/steps)
- (decision) MoSCoW COULD-HAVE (later): barcode scanning; voice food logging; meal planning/grocery/recipe nutrition
- (insight) SYNTHESIS: (1) Chips + ingredient-decomposition + DB-then-AI fallback = one pipeline: parse->decompose->match DB->ask chip only about uncertain proportion->confirm/save. (2) Curated DB is BOTH the trust 'source' and the offline cache - one asset kills accuracy fear AND speed fear. (3) In-vs-out dashboard is the retention/behavior-change engine ('make him thinking'), not just a readout. (4) MVP discipline: AI as calc engine now, conversational coach later. Chosen direction = build accurate-data-first tracker, profile as step zero.
